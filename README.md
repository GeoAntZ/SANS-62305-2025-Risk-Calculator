Technical Blueprint for IEC 62305-2:2024 Compliant Lightning Risk Assessment SoftwareThe development of professional-grade software for lightning risk assessment is governed by the rigorous demands of international safety standards, specifically the newly released IEC 62305-2:2024, which has been adopted in South Africa as SANS 62305-2:2025. A compliant program must serve as more than a simple calculator; it must function as a sophisticated risk management system that accounts for the complex interplay between environmental variables, structural geometry, electrical service characteristics, and human presence. The structural integrity of the software's codebase must mirror the modularity of the standard itself, ensuring that each mathematical annex—from the determination of dangerous event frequencies to the calculation of residual voltage probabilities for surge protective devices—is encapsulated in a verifiable and maintainable logic layer. This comprehensive architectural blueprint identifies the requisite file structure, data models, and algorithmic logic paths necessary to build a tool that transitions from manual, error-prone spreadsheets to a precise, data-driven engineering platform.Evolutionary Context of Edition 3 ComplianceThe transition from the previous 2010 edition to the 2024 revision represents a significant shift in the methodology of lightning risk evaluation. A critical requirement for any new software project is the implementation of the unified risk concept, which now integrates loss of human life and structural fire risks into a single, comprehensive risk category $R$. This change simplifies the final decision-making process for facilities managers but complicates the underlying software logic, as the calculation engine must now process overlapping causes of damage that were previously isolated. Furthermore, the introduction of "frequency of damage" ($F$) as a standalone metric for the availability of internal systems requires a dedicated logic branch in the software architecture to evaluate the reliability of data centers and healthcare facilities independent of physical structural damage.The most fundamental change at the environmental data layer is the replacement of lightning ground flash density ($N_G$) with lightning ground strike-point density ($N_{SG}$). For software developers, this means the input schema for geographic data must now support strike-point metrics, often requiring a conversion factor $k$ where $N_{SG} = k \cdot N_G$ if direct LLS (Lightning Location System) strike-point data is unavailable. These changes necessitate an architectural approach that is both flexible and version-aware, allowing users to select the appropriate edition of the standard for retrospective audits or new project designs.Software Architectural Philosophy: Modular Domain-Driven DesignTo manage the inherent complexity of over 70 parameters defined in the standard, the software should be constructed using Domain-Driven Design (DDD) principles. This approach ensures that the code's "ubiquitous language" aligns perfectly with the engineering terminology found in SANS 62305-2:2025, facilitating verification by subject matter experts. The architecture is divided into clear bounded contexts: the Environment Context handles geographic and topographical data; the Physical Configuration Context manages the geometric representation of the structure; and the Risk Assessment Engine implements the core mathematical logic.A modular architecture is paramount for this project because it enables "modular certification," where specific algorithm modules can be unit-tested against the benchmark case studies provided in Annex F of the standard. By decoupling the calculation logic from the user interface and the reporting engine, the software becomes resilient to future updates in the standard. This resilience is particularly important given that Edition 3 now recognizes Thunderstorm Warning Systems (TWS) as a legitimate protective measure, requiring the software to integrate real-time or probabilistic data from TWS sensors into the risk reduction equations.Architectural ComponentResponsibilityRelevant Standard SectionEnvironment ModuleCalculation of $N_D, N_M, N_L, N_I$ based on $N_{SG}$Annex AGeometry EngineDetermination of Collection Areas $A_D, A_M, A_L, A_I$Clauses A.2 - A.5Probability EngineProcessing of $P_A$ through $P_Z$ and $P_{SPD}$Annex B, Annex DLoss CalculatorDetermination of $L_1, L_2, L_3$ mean relative lossAnnex CRisk AggregatorSynthesis of total risk $R$ and frequency $F$Clause 6, 8, and 9Optimization LayerAI-driven selection of protection measuresClause 7.3Comprehensive Directory and File Structure BreakdownThe proposed file structure for the Lightning Risk Assessment (LRA) program is designed to be language-agnostic but is ideally suited for Python or C++, leveraging a layered architecture to separate concerns. Algorithmic Logic and Data Modeling for Annex A: Dangerous EventsThe program's logic begins with the assessment of the average annual number of dangerous events $N$. The primary input is the strike-point density $N_{SG}$. The software must implement a geometry processor to compute collection areas, which represent the equivalent surface area from which a lightning strike would be intercepted by the structure or its connected lines.Geographic and Topographical ModelingThe Environment model must store the following fields to satisfy the requirements of Clause A.2:VariableSymbolPurposeRange/TypeGround Strike Point Density$N_{SG}$Core frequency metricFloat ($strike-points/km^2/year$)Location Factor$C_D$Adjusts for topography/surroundings0.25 (lower objects) to 2.0 (hilltop)Environment Factor$C_E$Adjusts for shielding by other buildings0.01 (urban/tall) to 1.0 (rural)Conversion Factor$k$Relationship between $N_G$ and $N_{SG}$Default 2.0 The software's dangerous_events.py module uses these factors to compute $N_D$ (flashes to structure) using the equation:$$N_D = N_{SG} \times A_D \times C_D \times 10^{-6}$$Geometric Collection Area LogicA significant functional aspect of the structure file is the determination of $A_D$. For isolated rectangular buildings, the engine must implement the mathematical expression:$$A_D = (L \times W) + 2(3H)(L + W) + \pi(3H)^2$$For complex shapes, the software must support a graphical or multi-part logic. If a structure has elevated protrusions, the engine must calculate $A_D$ for the base height and $A'_D$ for the protrusion height $H_P$, selecting the maximum value to ensure a conservative risk profile. This logic is crucial for modern architectural designs that do not fit the standard "box" profile.Algorithmic Logic for Annex B: Damage Probability FactorsThe probability_logic.py file is the most granular component of the system, implementing the decision trees for 13 distinct probabilities. Each probability $P_X$ is a function of protective measures and site characteristics.Life Safety Logic ($P_A$ and $P_{AD}$)To calculate $R_{AT}$ (risk of shock from touch/step voltages), the software must evaluate:$$P_{AT} = P_{LPS} \times P_{am} \times r_t \times P_{TWS}$$The logic here requires an interactive UI where the user selects the LPS class (I through IV), which retrieves the corresponding $P_{LPS}$ value from Table B.3 (e.g., 0.02 for Class I, 0.2 for Class IV). If the user checks a "Warning Notice" box, $P_{am}$ is set to 0.1; if "Meshed Earth" is checked, $P_{am}$ becomes 0.01. The final $P_{AT}$ result is sensitive to the floor material, with $r_t$ values ranging from $10^{-2}$ for concrete to $10^{-5}$ for wood or linoleum.Physical Damage Logic ($P_B$ and $P_V$)The probability of fire or explosion ($P_B$) is driven by the structural material and fire-fighting provisions. The engine must implement a lookup for $P_S$ (dangerous sparking) where wooden structures are assigned 1.0 and continuous metal frameworks are assigned 0.5. This is then mitigated by $r_p$ (fire provisions), where automatic extinguishing systems reduce the risk by 80% ($r_p = 0.2$). A critical second-order insight for developers is that in Edition 3, $r_p$ must be set to 1.0 in zones containing lithium-ion batteries unless specific containment measures are proven, reflecting modern fire safety realities.Internal Systems Failure Logic ($P_C$ and $P_M$)For failures caused by the Lightning Electromagnetic Pulse (LEMP), the software must process the interaction between spatial shields ($K_{S1}$), internal wiring routing ($K_{S3}$), and SPD protection ($P_{SPD}$). The factor $K_{S3}$ is determined by the loop area of the wiring; unshielded cables with different routing (large loops) have a factor of 1.0, while cables in the same conduit (small loops) reduce this to 0.2.Implementation of Annex D: Advanced SPD Probabilistic ModelingOne of the most profound additions in Edition 3 is the detailed evaluation of $P_{SPD}$ in Annex D, which replaces the more generic values used in Edition 2. The annex_d_spd_evaluation.py module must implement a dual-probability check :Charge Probability ($P_Q$): The probability that the lightning current's charge exceeds the SPD's withstand capacity ($I_{imp}$). The software must integrate the log-normal distributions for both positive and negative first strokes as shown in Figure D.1.Protection Level Probability ($P_{Up}$): The probability that the residual voltage at the SPD port exceeds the equipment's impulse withstand voltage ($U_w$), accounting for voltage drops in connecting leads and inductive coupling in the circuit.The aggregated probability is calculated as:$$P_{SPD} = 1 - (1 - P_Q)(1 - P_{Up})$$This module requires the most significant computational resources, as it often involves reading from graphical data curves (Figures D.2 through D.14) to determine $P_{Up}$ as a function of lead length ($l_c$) and circuit width ($w$). Professional software must offer a "Simplified" mode using Table B.7/B.8 typical values and an "Advanced" mode for custom SPD selections.Data Modeling for Lines and SectionsThe logic for incoming lines (Source S3 and S4) must be recursive to handle the partitioning of lines into individual sections ($S_L$) as required by Clause 8.4.
